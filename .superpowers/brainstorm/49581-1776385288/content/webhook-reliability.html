Webhook Reliability Strategies

How to handle failed or missed webhooks

Your Concern: What if webhooks fail or get missed?

Real scenarios where webhooks can fail:
  • Your API is down during deployment
  • Network timeout between Kajabi and your server
  • Kajabi webhook system has a bug
  • Webhook payload is malformed and crashes your endpoint
  • Rate limiting or temporary blocking

Strategies for Webhook Reliability

A

Webhooks + Lightweight Reconciliation

Primary: Kajabi webhooks for real-time sync

Backup: Daily reconciliation job that only checks for discrepancies (not full poll)

How reconciliation works:
  1. Query Kajabi for members updated in last 48 hours
  2. Compare against your database records
  3. Only process differences (missed webhooks)
  4. Log discrepancies for investigation
Pros:
  • 99% of updates are real-time
  • Catches missed webhooks within 24 hours
  • Lower API usage than full polling
  • Reasonable complexity
Cons:
  • Still need idempotent processing
  • Potential for 24-hour sync delay
  • Two code paths to maintain
B

Webhooks + Dead Letter Queue + Manual Trigger

Primary: Kajabi webhooks with retry logic

Monitoring: Failed webhooks go to dead letter queue (DLQ)

Recovery: Admin UI has "Re-sync Member" button for manual fix

How it works:
  1. Webhook endpoint logs all attempts (success/failure)
  2. Failed webhooks stored in DLQ with retry metadata
  3. Dashboard shows DLQ count (alerts if > 0)
  4. Admin can manually trigger re-sync for specific member
  5. Optional: Auto-retry DLQ items every 6 hours
Pros:
  • Real-time for normal cases
  • Explicit visibility into failures
  • Manual override for critical issues
  • Simpler than full polling
Cons:
  • Requires monitoring dashboard
  • Admin intervention needed for failures
  • Silent failures if DLQ not monitored
  • No automatic catch-up
C

Full Hybrid (Webhooks + Hourly Polling)

Primary: Kajabi webhooks for real-time sync

Backup: Hourly polling for all changes in last 2 hours (overlap window)

How it works:
  1. Webhooks process changes immediately
  2. Every hour: poll Kajabi for changes in last 2 hours
  3. Idempotent processing (same change can be processed twice)
  4. Overlap window ensures no gaps
Pros:
  • Maximum reliability
  • Auto-recovery from failures
  • Fast catch-up (1 hour max delay)
  • No manual intervention needed
Cons:
  • Highest API usage
  • Most complex to implement
  • Duplicate processing overhead
  • Hardest to debug
D

Webhooks Only (Trust Kajabi)

Primary: Kajabi webhooks with standard retry

Backup: None - assume Kajabi webhook system is reliable

How it works:
  1. Kajabi sends webhook, you process it
  2. Kajabi retries failed webhooks (if they support it)
  3. Log all webhook attempts for debugging
  4. Monitor logs for errors
Pros:
  • Simplest implementation
  • Lowest API usage
  • Single code path
  • Easy to understand and debug
Cons:
  • No safety net for missed webhooks
  • Depends on Kajabi reliability
  • Silently drift if webhooks fail
  • Manual investigation required

Recommended Approach

Start with Option A (Webhooks + Daily Reconciliation)

This balances reliability with complexity:

  • ✅ Real-time sync for 99% of changes
  • ✅ Automatic catch-up within 24 hours
  • ✅ Reasonable API usage (1 request/day vs 24/day for hourly polling)
  • ✅ Can start with just webhooks, add reconciliation later
  • ✅ Easy to upgrade to Option C if 24-hour delay isn't acceptable
