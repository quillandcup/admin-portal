Intent-Based Reconciliation Architecture

App stores desired state, reconciles external systems to match

Core Philosophy

Admin Portal = Source of Intent

The app stores what should be true, then orchestrates external systems to match that reality.

Hiatus Flow (Stripe-Based Auto-Resume)

1️⃣ Admin Sets Hiatus

Admin Portal
  • Store: hiatus_start_date
  • Store: hiatus_end_date
  • Store: hiatus_reason
  • Status: on_hiatus
→

2️⃣ Orchestrate External Systems

Stripe API
  • Call: pause_subscription(days=X)
  • Stripe auto-resumes after X days
  • Stripe emails member before resume
Slack API
  • Remove from all channels
  • Store channels for re-invite

3️⃣ Stripe Auto-Resumes (No Action Needed)

Stripe
  • Auto-resume on hiatus_end_date
  • Charge member's card
  • Webhook: subscription.resumed
→
Admin Portal Webhook Handler
  • Update status: active
  • Clear hiatus dates
  • Re-invite to Slack channels

Email Aliasing Strategy

Problem: Members can change email in Kajabi or Slack independently

Solution: Multi-email identity mapping

Database Schema: 
members {
  id: uuid
  primary_email: text  -- Current "canonical" email
  name: text
  ...
}

member_email_aliases {
  member_id: uuid → members(id)
  email: text (unique)
  source: 'kajabi' | 'slack' | 'manual'
  is_primary: boolean
  added_at: timestamp
}
How it works:
  1. Kajabi webhook: member.updated → Add new email to aliases, update primary
  2. Slack Events: user.change → Add new email to aliases
  3. Member lookup: Check all aliases, not just primary email
  4. Reconciliation: Merge members if aliases overlap

Failure Handling: Retryable Idempotent Operations

When Slack API fails removing member from channels:

Operation Log Schema: 
sync_operations {
  id: uuid
  member_id: uuid
  operation_type: 'hiatus_start' | 'hiatus_end' | 'cancel' | ...
  desired_state: jsonb  -- What should be true
  steps: jsonb[]        -- [
    {action: 'pause_stripe', status: 'completed', attempt: 1},
    {action: 'remove_slack', status: 'failed', attempt: 3, error: '...'}
  ]
  overall_status: 'in_progress' | 'completed' | 'failed'
  created_at: timestamp
  updated_at: timestamp
}
Execution Flow:
  1. Start: Create sync_operation record
  2. Execute each step: Call API, log result
  3. On failure: Mark step as failed, set overall_status = 'failed'
  4. Retry job: Every 15 min, retry failed operations (up to 5 attempts)
  5. Dashboard alert: Show failed ops count, link to detail page
  6. Manual action: Admin can retry individual steps or mark as resolved
Idempotency: Each step checks current state before acting
  • pause_stripe: Check if already paused before calling API
  • remove_slack: Check if member is in channel before removing
  • invite_slack: Check if already invited before sending invite
