Channel Access Rules Configuration

How should we define which members get access to which Slack channels?

Example Rules From Your Use Cases

Trigger Channel(s) Action
Trial member converts to paid #members-only, #accountability Grant access
Member reaches 30 days #advanced-topics Grant access
Member joins 180 Program #180-program, #180-cohort-2026 Grant access
Member starts hiatus ALL channels Remove access
Member cancels ALL channels Remove access

How Should These Rules Be Defined?

A

Hard-Coded in Application

Rules written directly in TypeScript code

Example code: 
if (member.status === 'trial_converted') {
  await addToChannels(['members-only', 'accountability']);
}
if (member.days_since_join === 30) {
  await addToChannels(['advanced-topics']);
}
if (member.programs.includes('180')) {
  await addToChannels(['180-program']);
}
Pros:
  • Simple to implement
  • Fast execution
  • Type-safe
  • Easy to debug
 Cons:
  • Requires code deploy to change rules
  • Not visible to non-developers
  • Hard to audit what rules exist
B

Database Rules Engine

Rules stored in database tables, interpreted at runtime

Example schema: 
channel_access_rules {
  id: uuid
  trigger_type: 'status_change' | 'milestone' | 'program_join'
  trigger_value: jsonb  -- {from: 'trial', to: 'paid'}
  channels: text[]      -- ['members-only', 'accountability']
  action: 'grant' | 'revoke'
  active: boolean
}
Pros:
  • Change rules without deploy
  • Admin UI to manage rules
  • Easy to audit and report
  • Very flexible
 Cons:
  • Complex to implement
  • Slower execution
  • Harder to debug
  • Can create invalid rules
C

Configuration File (YAML/JSON)

Rules defined in a config file checked into git

Example config: 
channel_access_rules:
  - trigger: trial_converted
    channels: [members-only, accountability]
    action: grant

  - trigger: days_since_join >= 30
    channels: [advanced-topics]
    action: grant

  - trigger: program_joined(180)
    channels: [180-program, 180-cohort-2026]
    action: grant
Pros:
  • Version controlled (git history)
  • Readable by non-developers
  • Easy to review in PRs
  • Moderately flexible
 Cons:
  • Still requires deploy to change
  • No runtime editing
  • Syntax errors possible
D

Hybrid: Core Rules + Configurable Mappings

Hard-code the logic, configure the channel mappings

Example: 
// Code defines WHEN to grant access
if (member.status === 'trial_converted') {
  await grantAccessByTrigger('trial_converted');
}

// Database defines WHICH channels
program_channel_mappings {
  program_name: '180 Program'
  channels: ['180-program', '180-cohort-2026']
}

milestone_channel_mappings {
  milestone: 'days_30'
  channels: ['advanced-topics']
}
Pros:
  • Balance of flexibility and simplicity
  • Change channel names without deploy
  • Core logic is type-safe
  • Easy to add new programs
 Cons:
  • Can't add new trigger types without deploy
  • Two places to look (code + DB)
  • Medium complexity
Recommendation: Start with Option A (hard-coded) for MVP, migrate to Option D (hybrid) once you have 3+ programs with channel mappings. Option A is fastest to ship, Option D scales better long-term.
