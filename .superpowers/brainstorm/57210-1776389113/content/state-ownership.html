The Key Question: Hiatus Management

Hiatus is your primary use case. Where should the hiatus schedule live?

A

Admin Portal Owns Hiatus Schedule

Flow: Admin sets hiatus dates in portal → Portal pauses in Kajabi → Portal auto-resumes when date arrives

How it works:
  1. Admin Portal stores: hiatus_start_date, hiatus_end_date
  2. On start date: Portal calls Kajabi API to pause subscription
  3. On end date: Portal calls Kajabi API to resume subscription
  4. Portal handles all Slack channel removals/re-invites
Pros:
  • Portal is single source of truth for hiatus
  • Can track hiatus history easily
  • Can see upcoming hiatus end dates
  • Auto-resume logic lives in one place
Cons:
  • Need scheduled job for auto-resume
  • If member manually resumes in Kajabi, creates conflict
  • Portal must reconcile external changes
B

Kajabi Owns Subscription State, Portal Orchestrates

Flow: Admin sets hiatus in portal → Portal pauses in Kajabi + stores metadata → Kajabi webhook tells portal when resumed

How it works:
  1. Admin Portal stores: hiatus_reason, expected_resume_date (metadata only)
  2. Portal immediately pauses subscription in Kajabi
  3. Kajabi handles subscription state (paused/active)
  4. Portal listens to Kajabi webhooks for state changes
  5. Auto-resume: Portal calls Kajabi API on scheduled date
Pros:
  • Kajabi always has correct billing state
  • Easier to handle external changes (member resumes in Kajabi)
  • Portal metadata adds context (why/when)
  • Clear separation of concerns
Cons:
  • State split between two systems
  • Need to sync Kajabi state back to portal
  • Auto-resume still needs scheduled job

Conflict Scenarios

What should happen in these edge cases?

Scenario 1: Admin sets hiatus, member cancels directly in Kajabi
  • Option A: Cancellation wins - clear hiatus, remove from Slack
  • Option B: Block cancellation in Kajabi (disable self-service)
  • Option C: Detect cancellation, notify admin, ask what to do
Scenario 2: Member updates email in Kajabi, has active Slack invite
  • Option A: Auto-remove old email from Slack, invite new email
  • Option B: Keep Slack as-is, show warning in admin portal
  • Option C: Prevent email changes in Kajabi (require admin portal)
Scenario 3: Slack API fails when trying to remove member from channels
  • Option A: Fail the entire operation (rollback hiatus in Kajabi)
  • Option B: Log error, queue for retry, continue with hiatus
  • Option C: Mark member as "sync failed", notify admin
